Autoevaluación del Trabajo Práctico

Descargo de responsabilidad: Una revisión no es lo mismo que una corrección.

Recuerde que debe obtener EXCELENTE en los 5 items, en caso contrario, el trabajo estará desaprobado.

Consigna del Trabajo Práctico

El objetivo de este trabajo práctico es desarrollar una aplicación web funcional que integre HTML semántico, CSS con diseño responsive y JavaScript para gestionar funcionalidades dinámicas.

  • Debe haber una vista optimizada para desktop y una vista específica para dispositivos móviles.
  • El JavaScript debe manejar un CRUD completo (crear, leer, actualizar y eliminar datos).
  • El funcionamiento del sitio debe ser real y verificable: se valora que el código funcione, no las intenciones.
  • Sin embargo, se revisa el código para detectar buenas prácticas, orden, semántica y reutilización.
  • El repositorio en GitHub y la página desplegada deben estar correctamente configurados y accesibles.
  • El plantel docente puede convocar a una defensa oral o escrita por cualquier motivo si lo considera necesario.
Criterio Insuficiente (1) Regular (2) Bueno (3) Excelente (4)
HTML Estructura básica incorrecta; etiquetas esenciales no aplicadas adecuadamente. HTML con estructura básica y etiquetas principales, pero sin jerarquía o uso de semántica. HTML estructurado con jerarquía clara; etiquetas semánticas básicas aplicadas correctamente. HTML bien organizado y semántico, con estructura clara y optimización del código.
CSS Estilos desorganizados; sin diseño estructurado ni uso adecuado de propiedades. Estilos básicos con diseño funcional, pero falta de estructura o consistencia. CSS organizado con un diseño responsive para dispositivos móviles y de escritorio. CSS optimizado con diseño avanzado y responsivo, usando Flexbox o Grid según necesidad.
JavaScript Falta de interactividad o funciones básicas; código sin estructura. Funciones básicas y mínima interactividad, pero sin modularidad o reutilización de código. Código organizado, con funciones reutilizables y buenas prácticas, pero sin estructura avanzada. Código modular y bien organizado, con un manejo óptimo de lógica y funcionalidades CRUD completas.
Responsividad No hay adaptación a dispositivos móviles o de escritorio. Adaptación parcial; problemas de visualización en dispositivos móviles o de escritorio. Diseño responsivo que se adapta a dispositivos móviles y de escritorio con algunos ajustes necesarios. Diseño completamente adaptado y optimizado para dispositivos móviles y de escritorio sin problemas de visualización.
Usabilidad Navegación confusa, diseño desorganizado y falta de coherencia visual. Navegación funcional pero no intuitiva; diseño básico y limitado en usabilidad. Navegación clara y lógica; diseño visual con coherencia y buena experiencia de usuario. Navegación altamente intuitiva y eficiente, con un diseño visual atractivo que facilita la usabilidad.

El repositorio de GitHub está disponible, pero la página en producción no es accesible, o viceversa. Además, las fechas de los commits no son consistentes o no cumplen con las pautas. Se puede acceder al repositorio de GitHub y a la página en producción, pero las fechas de los commits no corresponden a un desarrollo continuo o presentan inconsistencias. Se puede acceder al repositorio de GitHub y a la página en producción (GitHub Pages o Vercel). Las fechas de los commits reflejan un proceso de desarrollo continuo y alineado a las pautas establecidas.
